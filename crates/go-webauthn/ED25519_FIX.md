# Ed25519 Go 1.25 FFI Crash Fix

## Problem

The Ed25519 signing implementation was crashing with a fatal error in Go 1.25:

```
fatal error: getWeakHandle on invalid pointer
runtime.throw({0x5ae7087bdefd?, 0x5ae708ac568f?})
```

### Root Cause

Go 1.25 introduced weak pointers in `crypto/internal/fips140cache` to optimize Ed25519 operations. The FIPS cache uses weak pointers to track private keys and avoid redundant computations.

**The Issue**: Weak pointers only work with Go-managed memory (allocated by Go's garbage collector). When we pass byte slices from Rust to Go through rust2go FFI, the memory is allocated in Rust's heap, not Go's heap. When `ed25519.Sign()` tries to cache the private key using a weak pointer, it fails because the memory isn't tracked by Go's GC.

## Solution

Make a **defensive copy** of the private key from FFI memory into Go-managed memory before passing it to `ed25519.Sign()`:

```go
func (c *CryptoImpl) ed25519_sign(req *Ed25519SignRequest) Ed25519SignResponse {
    // Verify private key size
    if len(req.private_key) != ed25519.PrivateKeySize {
        return Ed25519SignResponse{
            success: false,
            error:   fmt.Sprintf("invalid private key size: expected %d, got %d", 
                                ed25519.PrivateKeySize, len(req.private_key)),
        }
    }

    // Make a defensive copy of the private key from FFI memory
    // This avoids potential issues with weak pointers in Go 1.25 FIPS cache
    privateKeyCopy := make(ed25519.PrivateKey, len(req.private_key))
    copy(privateKeyCopy, req.private_key)

    // Sign the message with the Go-managed copy
    signature := ed25519.Sign(privateKeyCopy, req.message)

    return Ed25519SignResponse{
        signature: signature,
        success:   true,
        error:     "",
    }
}
```

### Key Points

1. **Allocate in Go**: `make(ed25519.PrivateKey, len(req.private_key))` allocates memory in Go's heap
2. **Copy data**: `copy(privateKeyCopy, req.private_key)` copies bytes from Rust FFI memory to Go memory
3. **Use Go copy**: Pass `privateKeyCopy` (not `req.private_key`) to `ed25519.Sign()`

## Performance Impact

**Minimal**: Copying a 64-byte private key is negligible compared to the Ed25519 signing operation itself (which involves elliptic curve math).

- Memory copy: ~64 bytes = **< 1 microsecond**
- Ed25519 signing: **~50-100 microseconds** on modern CPUs

The copy represents less than 2% overhead.

## When to Apply This Pattern

Use defensive copying for any Go crypto operation that:
1. Uses internal caching (FIPS cache, memoization)
2. Takes data from FFI/C/unsafe memory
3. May use weak pointers or finalizers

### Safe Operations (No Copy Needed)

- **ChaCha20-Poly1305**: No internal caching, operates purely on input data
- **Simple hash functions**: SHA-256, BLAKE2b (stateless)
- **Random number generation**: crypto/rand.Reader

### Operations That Need Copies

- **Ed25519**: FIPS cache uses weak pointers
- **ECDSA**: May use similar caching in future Go versions
- **RSA**: Uses caching for precomputed values
- **AES-GCM with hardware acceleration**: May cache key schedules

## Verification in Standalone Go

To verify Ed25519 works in Go without FFI:

```bash
cat > /tmp/test_ed25519.go << 'EOF'
package main

import (
    "crypto/ed25519"
    "crypto/rand"
    "fmt"
)

func main() {
    pub, priv, _ := ed25519.GenerateKey(rand.Reader)
    message := []byte("test")
    sig := ed25519.Sign(priv, message)
    valid := ed25519.Verify(pub, message, sig)
    fmt.Printf("Valid: %v\n", valid)
}
EOF
go run /tmp/test_ed25519.go
```

This works fine because the key is allocated in Go's heap from the start.
